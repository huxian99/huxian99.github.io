设计模式之装饰者模式(Decorator)

发表于 2015-05-06   |   分类于 设计模式   |  

装饰者模式定义：

动态地将责任附加到对象上。想要扩展功能，装饰者提供有别于继承的另一种选择。

UML类图：

使用场景：

  1. 需要扩展一个类的功能，或给一个类添加附加职责。
  2. 需要动态的给一个对象添加功能，而且还可以动态的撤销。(继承不能做到这一点，继承的功能是静态的，不能动态增删）
  3. 需要增加由一些基本功能的排列组合而产生的非常大量的功能，从而使继承关系变的不现实。

优缺点：

优点：

  1. 装饰者模式可以提供比继承更多的灵活性。
  2. 通过使用不同的具体装饰者类，我们可以创造出很多不同行为的组合。

缺点：

  1. 设计中会出现许多相似的对象，如果过度使用，会使程序变得很复杂，不容易排错。
  2. 装饰模式是针对抽象组件（Component）类型编程。但是，如果你要针对具体组件编程时，就应该重新思考你的应用架构，以及装饰者是否合适。当然也可以改变Component接口，增加新的公开的行为，实现“半透明”的装饰者模式。在实际项目中要做出最佳选择。

简单Demo：

非诚勿扰节目组考虑将24位女嘉宾的坐标，性格/爱好显示给用户。为此他们打算设计一套系统，你看到了这个信息，心想着帮他们设计一下。于是赶紧写起了代码：

public abstract class Girl{
    String description = "no particular";
    public String getDescription();
}

然后24个女嘉宾只需要继承这个抽象的Girl不就妥妥地ok了么。
看到一号女嘉宾是个很可爱的中国妹子: 

public class ChinaWithCute extends Girl {
    public String getDescription() {
        return "Chinese girl cute";
    }
}

看到二号女嘉宾是个很年轻的米国妹子: 

public class AmericaWithYoung extends Girl {
    public String getDescription() {
        return "America girl young";
    }
}

看到三号女嘉宾是个很善良的中国妹子: 

public class ChinaWithVirtuous extends Girl {
    public String getDescription() {
        return "China girl virtuous";
    }
}

看到四号女嘉宾，五号女嘉宾… 等等，这么说来要写24个妹子了！

旁边的小伙伴看到你在设计这个问题，于是略带嘲笑的问了问你：那如果一个女嘉宾走了，新来一个女嘉宾，你还要再实现一个妹子了？

聪明的你实在受不了自己这么糟糕的设计了，明明可以把这些性格/爱好抽象出来去装饰这些妹子么.

1. 妹子接口： 

public interface Girl {
    String description = "no particular";
    public String getDescription();
}

2. Concrete Component： 

/**
 * 中国妹子
 */
public class ChinaGirl implements Girl {

    @Override
    public String getDescription() {
        return "Chinese Girl";
    }

}

/**
 * 米国妹子
 */
public class AmericaGirl implements Girl {

    @Override
    public String getDescription() {
        return "American Girl";
    }

}

3. 装饰性格/爱好的接口： 

public interface GirlDecorator extends Girl {

}

4. Concrete Decorator： 

/**
 * 可爱的妹子
 */
public class Cute implements GirlDecorator {

    private Girl girl;

    public Cute(Girl girl) {
        this.girl = girl;
    }

    @Override
    public String getDescription() {
        return girl.getDescription() + " cute";
    }

}

/**
 * 孝顺的妹子
 */
public class Filial implements GirlDecorator {

    private Girl girl;

    public Filial(Girl girl) {
        this.girl = girl;
    }

    @Override
    public String getDescription() {
        return girl.getDescription() + " filial";
    }

}

/**
 * 善良的妹子
 */
public class Virtuous implements GirlDecorator {

    private Girl girl;

    public Virtuous(Girl girl) {
        this.girl = girl;
    }

    @Override
    public String getDescription() {
        return girl.getDescription() + " virtuous";
    }

}

/**
 * 年轻的妹子
 */
public class Young implements GirlDecorator {

    private Girl girl;

    public Young(Girl girl) {
        this.girl = girl;
    }

    @Override
    public String getDescription() {
        return girl.getDescription() + " young";
    }

}

4. 客户端测试： 

public class Test {

    public static void main(String[] args) {
        Girl china = new ChinaGirl();
        china = new Cute(china);
        china = new Filial(china);
        System.out.println(china.getDescription());

        Girl america = new AmericaGirl();
        america = new Virtuous(america);
        america = new Young(america);
        System.out.println(america.getDescription());
    }

}

result：
Chinese Girl cute filial
American Girl virtuous young

这时候新来了一个米国身材非常Hot的妹子，你可以动态地，更灵活的把她加入到你的系统中了吧。
